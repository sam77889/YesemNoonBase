## 502 Bad Gateway 修复（重大运维教训）
- 现象：用户通过 Cloudflare 访问报 502 Bad gateway Error code 502。
- 根因：本项目后端和代理由 **systemd 管理，且存在用户级和系统级两套同名服务**。用户级（~/.config/systemd/user/）实际提供服务。我之前手动 kill 用户级进程 + nohup 启动 uvicorn，破坏了用户级 systemd 服务管理：手动进程占 8001 → 用户级 noon-api restart 因 EADDRINUSE 失败循环 → 服务不可用 → cloudflared 连不上 → Cloudflare 502。
- 修复：kill 手动 nohup uvicorn 后，用户级 systemd 自动 restart 成功（noon-api PID 889029、noon-proxy PID 890055，均 active running 稳定）。502 消失。
- 遗留：系统级同名服务仍在 EADDRINUSE restart 循环（多余无害），建议 `sudo systemctl disable --now noon-api.service noon-proxy.service` 清理。
- 正确重启方式（已写入 MEMORY.md）：`systemctl --user restart noon-api.service` / `systemctl --user restart noon-proxy.service`，**绝不再手动 kill/nohup**。
- 验证注意：代理 SECURE_COOKIE=true + trust proxy=1，HTTP 直连无法登录（session cookie 不设置），真实 HTTPS 访问正常。

## 优化：fetcher 本地直连商品数从 7 提升到 45
- 现象：fetcher 搜 massage gun 一页只拿到 7 个商品，而 scraperapi 能拿 50。
- 根因：fetcher 原用 `extract_product_links`（DOM `<a>` 含 /p/+/N），但 noon 搜索页 SSR 的 DOM `<a>` 只有首屏 7 个，其余商品数据内联在 `__TSR__` JS 对象里（catalog_sku 出现 50 次），需 JS 渲染才会生成 `<a>`。curl_cffi 不执行 JS。
- 修复（fetcher_scraper.py）：`run_fetcher_scrape` 优先调用 `parse_noon_search_html`（复用 scraperapi 的 TSR 正则解析），一页拿全 50 个（含 price/rating/review_count，无需逐个抓详情页）；TSR 无数据时回退到 DOM 链接 + 详情页 JSON-LD 抓取。去重改用 sku 集合。
- 验证：massage gun 一页提取 50 个，清洗后入库 45 个（之前 7 个）。systemctl --user restart noon-api 生效。
- 教训：fetcher 与 scraperapi 的区别只在"抓 HTML 的方式"（curl_cffi vs ScraperAPI 代理），解析 noon 搜索页都应走 __TSR__ 正则。

## 优化：fetcher 单页抓取量从 50 提升到 200（noon limit 上限）
- 现象：用户反馈实际网页一页 80-90 个，fetcher 只拿 50。
- 调查：noon 搜索用 Algolia，SSR 数据含 `nbHits=1688`（总数）、`nbPages=34`、`limit:50,page:1`。每页 50 是默认。网页 80-90 是滚动加载第 2 页部分。
- 关键发现：noon 搜索 URL 支持 `&limit=N` 参数！实测 limit=100→100个，limit=200→200个，limit≥200 都返回 200（**200 是单次上限**，HTML 3.1MB）。
- limit=200&page=2 翻页有效（两批去重 394），所以 limit=200 + 翻页是最高效方案。
- 修复（fetcher_scraper.py）：search_url 加 `&limit=200`，翻页 URL 自动继承。
- 验证：massage gun 单页提取 200 个，清洗后入库 192 个（之前 50→清洗 45）。
- 单次抓取量演进：7（DOM链接）→ 50（TSR解析）→ 200（limit=200）。
